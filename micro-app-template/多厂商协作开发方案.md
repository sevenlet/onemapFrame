# 一张图框架 3.0 —— 多厂商协作开发方案

> **版本**：v1.0  
> **日期**：2026 年 6 月  
> **密级**：内部

---

## 一、背景与目标

### 1.1 背景

一张图平台作为数据大屏可视化展示的核心载体，需要集成多个业务厂商开发的功能模块（如环境监测、污染溯源、应急指挥、视频监控等）。这些模块由不同厂商独立开发、独立部署，最终统一加载到平台中运行。

当前面临的挑战：

- **厂商技术栈不统一**：各厂商使用的框架版本、组件库、构建工具各不相同，直接集成困难
- **耦合度高**：传统的 npm 包或源码合并方式，导致构建依赖交叉、版本冲突、发布互相阻塞
- **升级困难**：平台基础能力（地图引擎、全局数据）升级时，所有厂商需同步改代码、重新联调
- **安全边界模糊**：厂商代码直接运行在主应用上下文中，CSS 全局污染、JS 变量冲突、DOM 操作越界等问题频发

### 1.2 目标

| 目标 | 度量 |
|------|------|
| **开发解耦** | 各厂商独立开发、独立构建、独立部署，互不依赖 |
| **技术栈无关** | 厂商可使用任意前端技术栈（Vue/React/Angular/原生 JS） |
| **运行隔离** | 厂商模块间样式、脚本、DOM 完全隔离，互不干扰 |
| **集成简单** | 厂商只需在平台配置微应用地址即可上线，无需平台侧改代码 |
| **体验一致** | 全局数据、地图能力、弹窗等平台能力对厂商透明可用 |

## 二、方案选型

### 2.1 候选方案对比

| 维度 | iframe 原生嵌套 | NPM 包集成 | Module Federation | **微前端（micro-app）** |
|------|:---:|:---:|:---:|:---:|
| 开发解耦 | ✅ 完全独立 | ❌ 构建耦合 | ⚠️ 构建时部分耦合 | ✅ 完全独立 |
| 技术栈无关 | ✅ | ❌ 需统一 | ⚠️ 有限支持 | ✅ |
| 运行隔离 | ✅ 物理隔离 | ❌ 共享上下文 | ❌ 共享上下文 | ✅ 沙箱隔离 |
| 通信能力 | ⚠️ postMessage 繁琐 | ✅ 直接调用 | ✅ 共享模块 | ✅ 三通道 Bridge |
| 集成成本 | ⚠️ 需适配 | ❌ 改动大 | ⚠️ 配置复杂 | ✅ 配置地址即可 |
| 弹窗穿透 | ❌ 被框住 | ✅ | ✅ | ✅ RPC 弹窗穿透 |
| 地图共享 | ❌ 需重建 | ✅ | ⚠️ | ✅ 代理远程调用 |
| 样式隔离 | ✅ | ❌ | ❌ | ✅ CSS 沙箱 |

### 2.2 选型结论

**采用 @micro-zoe/micro-app 微前端方案**，核心考量：

1. **iframe 沙箱模式**：天然继承 iframe 的物理隔离优势，同时通过 Bridge 机制补齐了通信短板
2. **三通道通信**：定向数据下发、消息/RPC 调用、全局状态共享，覆盖所有协作场景
3. **使用成本低**：厂商只需下载工程模板，按 API 开发即可，无需理解底层机制
4. **生态成熟**：micro-app 在国内微前端领域应用广泛，社区活跃，文档完善

### 3.2 角色职责

| 角色 | 职责 | 产出 |
|------|------|------|
| **平台方** | 提供基座运行环境、工程模板、API 文档、联调支持 | 基座 + 模板 + 文档 |
| **厂商** | 在模板微应用中开发业务代码，独立构建部署 | 微应用构建产物 + 访问地址 |
| **运维方** | 在平台后台配置微应用名称与地址，完成上线 | 平台配置 |

### 3.3 三通道通信模型

基座与微应用之间通过三层通信通道协作，各通道职责明确、互不干扰：

```
                  基座（Base）                         微应用（Micro）
              ┌──────────────┐                   ┌──────────────┐
 通道① 定向   │  dataToChild │ ──── :data ────▶  │   baseData   │
  数据下发    │  setData()   │                   │  (只读响应)   │
              └──────────────┘                   └──────────────┘

              ┌──────────────┐                   ┌──────────────┐
 通道② 消息   │ registerHandler│ ◀── dispatch ───│  sendToBase() │
  与 RPC     │ registerMethod │ ◀─ callBase() ──│  callBase()  │
              │  → 返回结果    │ ── setData ──▶  │   (Promise)  │
              └──────────────┘                   └──────────────┘

              ┌──────────────┐                   ┌──────────────┐
 通道③ 全局   │  globalData  │ ◀──────────────▶  │  globalData  │
  状态共享    │setGlobalData │                   │setGlobalData │
              └──────────────┘                   └──────────────┘
```

| 通道 | 方向 | 典型场景 | 通信方式 |
|------|------|---------|---------|
| 通道① 定向数据 | 基座 → 微应用 | 下发用户信息、Token、主题 | `:data` 属性，微应用响应式接收 |
| 通道②a 消息 | 微应用 → 基座 | 通知基座行为、发送日志 | `dispatch`，基座注册 handler 处理 |
| 通道②b RPC | 微应用 ↔ 基座 | 调用基座方法并获取返回值 | `callBase()`，支持 Promise/Await |
| 通道③ 全局状态 | 双向 | 区域信息、主题、配置等共享 | `globalData`，双向可读写 |

### 4.2 各阶段详述

| 微应用（micro） | 厂商业务代码的开发容器，**最终交付物** |
| 通信 Bridge | 三通道 API 的完整封装，开箱即用 |
| 全局变量 & 函数 | 与生产环境一致的全局数据定义 |
| 示例页面 | 演示三通道用法、弹窗穿透、地图代理等 |
| API 文档 | 完整的开发者接口参考 |

**厂商只需关注 `packages/micro/src/` 目录下的代码开发。**

## 五、核心能力

### 5.1 全局数据共享

基座将平台核心数据（行政区信息、服务地址、主题配置、Socket 配置等）通过 globalData 通道同步给所有微应用，厂商可读取和修改：

```javascript
// 读取全局数据
import { useGlobalData, useGlobalDataKey } from '@/bridge.js'

// 方式一：获取全部
const { globalData } = useGlobalData()
console.log(globalData.value.regionaQuery?.regionCode)

// 方式二：订阅单个字段（推荐）
const regionCode = useGlobalDataKey('regionaQuery.regionCode')
watch(regionCode, (newVal) => {
  console.log('区域变了:', newVal)
})

// 修改全局数据（合并式，不会覆盖其他字段）
import { setGlobalData } from '@/bridge.js'
setGlobalData({ activeTheme: '蓝色版' })
```

**主要全局数据一览**：

| 分类 | 字段 | 说明 |
|------|------|------|
| 行政区 | `regionaQuery.regionCode` / `regionName` | 当前选中区域编码与名称 |
| 服务地址 | `serviceBaseUrl` | 主服务 API 地址 |
| 通信 | `socketIp` / `socketPath` / `webSocketUrl` | WebSocket 连接配置 |
| 主题 | `activeTheme` | 当前主题标识 |
| 会话 | `socketRoom` | 通信房间号 |
| 应用 | `appId` | 应用标识 |

### 5.2 远程方法调用（RPC）

微应用可通过 `callBase()` 调用基座注册的方法，获取返回值：

```javascript
import { useBaseBridge } from '@/bridge.js'
const { callBase } = useBaseBridge()

// 获取 Token
const token = await callBase('getToken')

// 弹出基座弹窗（穿透 iframe，覆盖整个浏览器）
const result = await callBase('showDialog', {
  componentName: 'RegionPickerDialog',  // 基座注册的弹窗组件名
  title: '请选择区域',
  props: { regions: [...] }
})

// 调用基座全局函数（自动 fallback 到 window 上的函数）
const urlParam = await callBase('getUrlQueryString', 'key')
```

**内置 RPC 方法**：

| 方法名 | 参数 | 返回值 | 说明 |
|--------|------|--------|------|
| `showDialog` | `{ componentName, title?, width?, props? }` | `Promise<any>` | 弹出基座弹窗 |
| `showToast` | `message: string` | — | 弹出基座提示 |
| `navigate` | `path: string` | — | 请求基座导航 |
| `getToken` | — | `string` | 获取当前 Token |
| `tmapCall` | `{ method, params }` | `Promise<any>` | 调用地图 API |

### 5.3 地图代理

微应用通过 ES Proxy 代理，可像直接操作地图一样调用基座的地图 API：

```javascript
import { TMap, createLayer } from '@/ths-map.js'

// 直接调用地图方法（自动转发到基座）
TMap.goTo({ center: { x: 114, y: 26 } })
TMap.setZoom(12)
TMap.addMarker({ lng: 114, lat: 26, title: '监测点' })

// 创建图层代理
const pointLayer = createLayer('pointLayer')
pointLayer.show()
pointLayer.setData([...])
```

**原理**：所有 TMap 方法调用通过 `callBase('tmapCall', { method, params })` 转发到基座，由基座操作真实地图实例，结果通过 Promise 回传。

### 5.4 弹窗穿透 iframe

微应用运行在 iframe 沙箱中，常规弹窗（如 Element Plus 的 ElDialog）被限制在 iframe 边界内。通过 RPC 弹窗机制，可实现弹窗覆盖整个浏览器：

```
┌─────────────────────────────────────────────┐
│  浏览器窗口                                   │
│  ┌───────────────────────────────────────┐   │
│  │          基座（生产环境）                │   │
│  │  ┌─────────────────────────────────┐  │   │
│  │  │      微应用（iframe 沙箱）       │  │   │
│  │  │                                 │  │   │
│  │  │  callBase('showDialog')  ──────│──│──│──▶ 基座渲染弹窗
│  │  │                                 │  │   │
│  │  └─────────────────────────────────┘  │   │
│  │                                       │   │
│  │  ┌─ 弹窗（基座渲染，覆盖整个浏览器）──┐  │   │
│  │  │                                   │  │   │
│  │  │   ✓ 穿透 iframe，全屏覆盖          │  │   │
│  │  │   ✓ 不被裁切                       │  │   │
│  │  │   ✓ 可返回结果给微应用              │  │   │
│  │  │                                   │  │   │
│  │  └───────────────────────────────────┘  │   │
│  └───────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

### 5.5 基座注入（provide/inject）

基座通过 Vue 的 `provide('root')` 向微应用注入核心能力（需在嵌入模式下通过 inject 获取）：

| 注入项 | 类型 | 说明 |
|--------|------|------|
| `rootEmit` | `Function` | 向基座发送事件 |
| `rootOn` | `Function` | 监听基座事件 |
| `rootOff` | `Function` | 取消监听 |
| `rootSocket` | `Socket` | Socket.io 实例（只读） |
| `setSocketRoomId` | `Function` | 设置通信房间号 |
| `sendMessage` | `Function` | 发送 WebSocket 消息 |

## 六、安全保障

### 6.1 运行时隔离

| 隔离维度 | 机制 | 效果 |
|----------|------|------|
| **JS 沙箱** | iframe 沙箱模式 | 各微应用 JS 上下文完全隔离，全局变量互不污染 |
| **CSS 隔离** | iframe 物理隔离 | 样式天然隔离，无需额外的 CSS Scope 方案 |
| **DOM 隔离** | iframe 独立 document | 微应用 DOM 操作不会影响基座和其他微应用 |
| **路由隔离** | 各自 hash 路由 | 微应用路由独立运行，不与基座路由冲突 |

### 6.2 通信安全

| 安全措施 | 说明 |
|----------|------|
| **RPC 方法白名单** | 基座仅响应 `registerMethod` 注册的方法，未注册方法不可调用 |
| **弹窗组件白名单** | 基座仅渲染 `dialogComponents` 注册表中的组件，防止任意组件注入 |
| **window fallback 受限** | `callBase` 的 window 回退仅支持全局函数，不支持属性读写 |
| **数据序列化** | 跨 iframe 通信仅支持可序列化数据，函数/DOM 等不可传递 |
| **超时机制** | RPC 调用 10 秒超时，防止恶意方法阻塞 |

### 6.3 部署安全

| 安全措施 | 说明 |
|----------|------|
| **跨域控制** | 微应用需配置 CORS 允许基座域名访问，否则加载失败 |
| **HTTPS** | 生产环境强制 HTTPS，防止中间人攻击 |
| **资源完整性** | 微应用构建产物可配置 SRI（Subresource Integrity）校验 |
| **内容安全策略** | 基座可配置 CSP 限制微应用资源加载范围 |

## 七、厂商开发规范

### 7.1 目录约定

```
packages/micro/src/
├── main.js              # 入口（已有，勿改）
├── App.vue              # 根组件（已有，勿改）
├── router.js            # 路由配置（按需修改）
├── bridge.js            # 通信桥（已有，勿改）
├── ths-map.js           # 地图代理（已有，勿改）
└── views/               # ← 厂商在此开发业务页面
    ├── Home.vue
    └── ...
```

### 7.2 开发红线

| 规则 | 说明 |
|------|------|
| ❌ 不修改 bridge.js | 通信层由平台统一维护，厂商不得修改 |
| ❌ 不修改 main.js | 入口文件由平台统一维护 |
| ❌ 不操作基座 DOM | 微应用不应通过 DOM API 操作 iframe 外的元素 |
| ❌ 不依赖基座内部实现 | 仅通过 Bridge API 与基座交互 |
| ✅ 使用 Bridge API 通信 | 数据读写、方法调用、弹窗等一律走 Bridge |
| ✅ 使用相对路径构建 | `base: './'`，确保部署后资源可加载 |
| ✅ 资源可序列化 | 跨 iframe 传递的数据必须可 JSON 序列化 |

### 7.3 独立运行兼容

微应用在脱离基座独立运行时（如本地直接访问 :5175），所有 Bridge API 自动降级：

| API | 嵌入基座时 | 独立运行时 |
|-----|-----------|-----------|
| `baseData` | 响应式接收基座数据 | 空对象 `{}` |
| `sendToBase()` | 消息发送到基座 | 控制台打印（仅调试） |
| `callBase()` | RPC 调用，返回结果 | `Promise.resolve(undefined)` |
| `globalData` | 响应式接收全局数据 | 空对象 `{}` |
| `setGlobalData()` | 合并更新全局数据 | 控制台打印（仅调试） |
| `useGlobalDataKey()` | 响应式订阅字段 | `ref(undefined)` |
| `TMap.xxx()` | 远程调用地图 API | `Promise.resolve(undefined)` |

## 八、集成与上线

### 8.1 上线检查清单

厂商提交上线前，需确认以下事项：

- [ ] 微应用构建产物可独立访问（HTML 正常加载）
- [ ] 微应用地址配置 CORS 允许基座域名访问（若跨域）
- [ ] 所有 Bridge API 调用在独立运行时不会报错
- [ ] 构建产物使用相对路径（`base: './'`）
- [ ] 无硬编码的 localhost 或内网地址
- [ ] 弹窗使用 `callBase('showDialog')` 而非 iframe 内弹窗（需穿透场景）
- [ ] 地图操作通过 `TMap` 代理调用，未直接操作地图实例

### 8.2 微应用接入配置

运维人员在平台后台完成以下配置：

```
{
  "name": "vendor-a-monitor",           // 微应用唯一标识
  "url": "https://vendor-a.com/app/",   // 微应用访问地址
  "iframe": true,                        // 使用 iframe 沙箱
  "enabled": true                        // 是否启用
}
```

**接入要求**：
- `name` 在平台内全局唯一
- `url` 必须可由用户浏览器直接访问
- 若 `url` 与基座不同源，微应用服务器需返回 CORS 响应头：

```
Access-Control-Allow-Origin: <基座域名>
```

## 九、方案优势总结

| 优势 | 具体收益 |
|------|---------|
| **零耦合开发** | 厂商之间、厂商与平台之间开发互不依赖，各自迭代 |
| **技术栈开放** | 厂商可自由选择前端框架，平台不设限 |
| **即插即用** | 配置地址即可上线，平台侧零代码改动 |
| **沙箱隔离** | iframe 模式天然隔离，无 CSS/JS/DOM 污染风险 |
| **能力共享** | 地图、弹窗、全局数据等平台能力对厂商透明可用 |
| **独立调试** | 厂商可脱离基座独立开发调试，降低联调成本 |
| **安全可控** | RPC 白名单 + 弹窗组件注册表 + 通信序列化，三重安全防线 |
| **平滑升级** | 平台基础能力升级时，只需更新模板和 Bridge API，厂商业务代码无需改动 |

### D. 术语表

| 术语 | 含义 |
|------|------|
| 基座 / Base | 一张图平台的运行容器，提供地图、全局数据等基础能力 |
| 微应用 / Micro | 厂商开发的功能模块，运行在基座的 iframe 沙箱中 |
| Bridge | 基座与微应用之间的通信桥接层 |
| RPC | Remote Procedure Call，微应用远程调用基座方法 |
| globalData | 全局共享数据通道，基座与所有微应用共用 |
| 沙箱 | micro-app 的 iframe 隔离模式，确保微应用互不干扰 |
| 弹窗穿透 | 微应用通过 RPC 让基座渲染弹窗，突破 iframe 边界限制 |
